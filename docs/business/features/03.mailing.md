# MAIL — Mailing

Email infrastructure for transactional flows. Today it powers password reset and stub-account activation through a single token model, with **Resend** as provider. Other features (auth, sheets) trigger emails through this layer; the user-facing UCs live in their respective features.

## Rules

### Provider & deliverability
- Resend is the email provider (free tier 3,000/month).
- DNS verification (SPF + DKIM) must be in place before any deploy that sends emails — without it, deliverability is effectively zero.

### Token model (shared across all token-based flows)
- A single table `password_reset_tokens` backs every token-based flow (reset password, stub activation, future activation-on-share).
- Tokens are generated with `crypto.randomBytes` and hashed (SHA-256) before storage. The raw value is never persisted.
- Exactly one active token per user — generating a new one invalidates previous tokens for the same user.
- Tokens expire 30 minutes after creation and are single-use (`used_at` set on consumption).
- Cleanup is lazy: expired tokens for a user are deleted when generating a new one. Periodic cleanup is on the backlog.

### Anti-enumeration & rate limiting
- Public endpoints that take an email return identical responses regardless of whether the email exists.
- Rate limit: 5 reset requests per hour per email.

### Sending
- Phase 1 emails are sent synchronously inside the request handler. Async/batch is on the backlog.

### Code layout
- Service wrapper: `server/services/email.ts` (Resend init + `sendEmail`).
- Templates: `server/templates/<flow>.ts`, one file per flow (`passwordReset.ts`, `accountActivation.ts`).

## Use cases

<details>
<summary>

### UC-MAIL-01 — Issue and dispatch a transactional token email · DONE

</summary>

- **Status:** DONE
- **Actor:** System (invoked by auth or sheet-sharing flows)
- **Trigger:** A flow needs to send a tokenized email to a user (e.g. password-reset request).
- **Refs:** AUTH

**Main flow**
1. Validate rate limit for the email (≤ 5/hour).
2. Invalidate previous tokens for the user (lazy cleanup of expired ones).
3. Generate a raw token + hash; store the hash with `expires_at = now + 30 min`.
4. Render the template for the flow (`passwordReset` / `accountActivation`).
5. Send via Resend with the link containing the raw token.
6. Return 200 generic to the caller (no signal about whether the email exists).

**Alternative flows**
- **1a. Rate limit exceeded:** caller receives a 429 (or the same generic 200, depending on the public endpoint contract — see UC-AUTH-04).

</details>

<details>
<summary>

### UC-MAIL-02 — Verify and consume a single-use token · DONE

</summary>

- **Status:** DONE
- **Actor:** System (invoked by `POST /api/auth/reset-password`)
- **Trigger:** A public endpoint receives a raw token + payload (e.g. new password).
- **Refs:** AUTH

**Main flow**
1. Hash the raw token and look it up.
2. Verify it exists, has not expired and `used_at IS NULL`.
3. Hand the resolved `user_id` back to the caller for whatever side effect the flow needs (set password, activate stub).
4. Mark `used_at = now`.

**Alternative flows**
- **2a. Invalid / expired / already-used token:** rejected with a generic error; the caller surfaces it without leaking which condition failed.

</details>

## Backlog

### Async/batch email job queue

Today email sending is synchronous on the request thread. Adds latency to `forgot-password` and any future endpoint that triggers emails, and offers no retries on Resend transient failures. We want to move sending to a queue with retries.

- [ ] Decide queue tech: BullMQ + Redis (extra deploy dependency) vs. SQLite-based queue (keeps the stack simple).
- [ ] Move reset-password and (future) share-activation emails to the queue.
- [ ] Retry policy + dead-letter handling.
- [ ] Fire-and-forget contract for callers — public endpoints stay non-blocking.

### Auto-activation email when sharing a sheet with a stub

Today, when a sheet is shared with a non-existing email, the invited person only finds out if the owner tells them and asks them to use the password-reset flow manually. UX gap. We want the system to send an activation email automatically at invite time so the invited user has a clear path in.

- [ ] Hook into `POST /api/sheets/:id/shares`: if the invited email is a stub, enqueue an activation email (UC-MAIL-01 path) with the existing token model.
- [ ] Decide token TTL for activation: reuse 30 min or extend.
- [ ] New template `accountActivation.ts` copy localized (es primary).
- [ ] Behavior if a previous activation email is still pending when the same email is invited to another sheet: invalidate-and-reissue (consistent with the "one active token per user" rule).
- [ ] Update sheets feature flow (UC-SHT-07) to reference the new auto-trigger.

### Periodic cleanup of expired tokens

Without a periodic job, expired tokens accumulate in SQLite. Lazy delete on issue keeps it bounded per active user but doesn't reclaim space for inactive users. Move from lazy-only to a scheduled cleanup once volume justifies it.

- [ ] Cron / queue-based job that deletes tokens with `expires_at < now - 24h`.
- [ ] Replace (or keep alongside) the current lazy delete on issue.
- [ ] Maintenance script `npm run db:clear-tokens` (`scripts/clearResetTokens.ts`) already exists — wire it as the job's underlying call.

### Forecast email delivery

Pairs with the forecast feature in [`features/01.expenses.md`](01.expenses.md) backlog. Once the signal and threshold are decided there, the actual delivery (template + scheduled send) lives here.

- [ ] Forecast email template.
- [ ] Scheduled job (cron or queue-based) that picks users above threshold and dispatches via UC-MAIL-01.
- [ ] Opt-in/opt-out preference per user (data model TBD).

## Out of scope (of the email service itself)

- Email verification at signup.
- Changing the user's email address.
- Admin dashboard / email analytics.
- Custom branding beyond functional HTML.
- OAuth account recovery (Google does not need a reset).
- 2FA codes.
- Push notifications (email-only for now).
