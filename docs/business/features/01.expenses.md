# EXP — Expenses

Core domain of the product: capture every spend line, classify it with per-user categories, list and filter by month, visualize monthly aggregates with totals and charts, and export the visible month to Excel. Categories and visualization live here because neither has standalone purpose — categories exist to classify expenses, visualization shows aggregates over them.

## Rules

- `amount > 0` for every expense.
- Each expense belongs to exactly one sheet and one category.
- Category name is unique per user (case-sensitive at the moment).
- 8 default categories are seeded on signup so the user can log expenses immediately.
- When a category is deleted, the system must keep historical expenses readable (handling TBD: reassign to a sentinel category vs. block deletion if in use).
- The expenses listing endpoint returns pre-aggregated data per month — totals, category breakdown and chart data come ready from the backend; the UI never re-aggregates from raw rows (no N+1 from the table render).
- The month-over-month delta compares the visible month with the immediately previous month on the same sheet.
- The category breakdown is sorted by amount descending.
- When expense payments ship, all aggregates here switch from gross `amount` to `effective = amount − SUM(payments)`.

### UC-EXP-14 — Move expense to another sheet/month · DONE

</summary>

- **Status:** DONE
- **Actor:** Owner user (or invited user with `edit` on the source sheet)
- **Trigger:** User opens the row menu on an expense and clicks "Mover".

**Main flow**
1. The row's kebab menu opens; user clicks "Mover".
2. `MoveExpenseDialog` opens with two fields, defaulted to the current sheet and `source.month + 1`. The sheet dropdown lists every sheet where the actor has edit permission (own + shared with `edit`).
3. User picks the destination sheet and month and submits. Same sheet + same month is allowed (treated as a no-op update).
4. Frontend computes the resolved date with `clampDayToMonth(targetYear, targetMonth, source.date.day)` (e.g., 31-Jan → 28/29-Feb) and submits `{ sheetId, date }`.
5. Backend updates the expense in place: `sheet_id = targetSheetId`, `date = resolved_date`. `id`, `created_at`, `user_id` and any future `expense_payments` references stay intact.
6. Aggregates of source month and destination month refresh; success toast "Movido a {sheet} ({mes})" appears with a "Ver" button that switches the active sheet + month to the destination.

**Alternative flows**
- **5a. Actor lacks `edit` permission on source or destination sheet:** 403; dialog closes with a generic error toast.

**Outcome**
- The expense's classification (sheet + month) changes; its identity (`id`) and history (`created_at`, `user_id`) do not. Linked `expense_payments` (when shipped) keep pointing at the same `expense_id`.

</details>

### UC-EXP-15 — Duplicate expense to another sheet/month · DONE

</summary>

- **Status:** DONE
- **Actor:** Owner user (or invited user with `edit` on the source sheet)
- **Trigger:** User opens the row menu on an expense and clicks "Duplicar".
- **Parents:** UC-EXP-14 (depends on the kebab refactor and the `clampDayToMonth` helper introduced there).

**Main flow**
1. The row's kebab menu opens; user clicks "Duplicar".
2. `DuplicateExpenseDialog` opens with the same shape as the Move dialog (sheet dropdown listing every sheet where the actor has edit permission), defaulted to the current sheet and `source.month + 1`.
3. User picks the destination sheet and month and submits. Same sheet + same month is allowed (creates an exact copy, no warning).
4. Frontend computes the resolved date with `clampDayToMonth` and submits `{ targetSheetId, date }`.
5. Backend inserts a new row: `amount`, `description`, `category_id` copied from source; `sheet_id = targetSheetId`; `date = resolved_date`; `user_id = req.userId` (the actor, not the source's `user_id`); fresh `id` and `created_at`. **No `expense_payments` copied.**
6. Aggregates of destination month refresh; success toast "Duplicado en {sheet} ({mes})" appears with a "Ver" button that switches the active sheet + month to the destination.

**Alternative flows**
- **5a. Actor lacks `edit` permission on source or destination sheet:** 403.

**Outcome**
- A new expense exists in the destination sheet/month with a fresh `id` and `user_id = actor`. The source row is unchanged.

<summary>

### UC-EXP-16 — Recurring expense templates with email alerts · PENDING

</summary>

- **Status:** PENDING
- **Actor:** Owner user (creates and manages templates) + System (materializes instances and sends emails)
- **Trigger:** User opens the "Recurrentes" modal on a sheet to automate a periodic expense.

**Main flow** — Create template
1. User clicks "Recurrentes" on the sheet header. A modal opens listing the actor's own templates on this sheet (active + inactive). Templates of other users on the same sheet are not shown — templates are private to the creator.
2. User clicks "+ Nueva" and fills the form: amount (>0), description, category (own categories), sheet (sheets where actor has `edit`, default = current), period (`mensual` | `anual`), start_date (first cobro date), notice_days (default 3, labeled "Avisarme con X días de antelación").
3. User submits. Backend persists a `recurring_expenses` row with `user_id = req.userId`, `is_active = true`, `last_generated_period_index = -1`. The next due date is computed as `start_date`.
4. The list refreshes; the new template appears with its next cobro date.

**System flow** — Materialize instance + email
5. A daily cron at 06:00 UTC walks active templates. For each one whose next due date is within `notice_days`:
   a. Re-checks the creator still has `edit` on `sheet_id`. If not, sets `is_active = false` and stops.
   b. Computes `resolved_date = clampDayToMonth(targetYear, targetMonth, start_date.day)` (reuses helper from UC-EXP-14).
   c. Inserts an expense row: `sheet_id`, `category_id`, `amount`, `description`, `user_id` copied from the template; `date = resolved_date`; fresh `id` and `created_at`. The row is a normal expense from this point on.
   d. Advances `last_generated_period_index` (same transaction as the insert).
   e. Sends an email to the creator: subject "Recordatorio: {description} - {amount}€ el {fecha}", body summarizes the upcoming charge with a link back to the sheet.
6. The new expense surfaces in the affected month's table and aggregates on the user's next visit.

**Alternative flows**
- **Edit template:** the creator edits any field. Changes apply only to **future** generations; already-generated instances are untouched (independent normal expenses).
- **Toggle active/inactive:** sets `is_active`. Future generation stops or resumes; existing instances are not modified. Reactivating does **not** backfill missed periods.
- **Delete template:** removes the `recurring_expenses` row (with confirm). Already-generated instances remain as orphan normal expenses.
- **User deletes a generated instance:** `last_generated_period_index` is **not** rolled back; the cron does not regenerate for that period.
- **Creator loses `edit` on the sheet** (sheet owner revokes the invite): cron auto-deactivates the template on the next run. If access is restored, the user can manually reactivate.

**Outcome**
- Templates are private to the creator; they are not visible to other users with access to the same sheet.
- Generated instances are normal expenses and follow the visibility/permission rules of the sheet they live on. When `expense_payments` ships, payments attach to instances, not to templates.
- Aggregates always read from materialized rows in `expenses`; there are no "virtual" or pending rows in monthly totals.
- Email is the only alert channel in v1; in-app and push are deferred.

**Future extensions** (out of scope for v1)
- Custom periodicity (`every N months/years`). Schema can migrate to `period_unit` + `period_interval` without rewriting rows.
- Unified sheet settings drawer (Categorías + Recurrentes + Compartir + General). v1 ships as a stand-alone modal and may be folded into a wider settings panel later.

## Backlog

### Expense payments (partial reimbursements)

Users buy things they share with others (dinners, gifts, trips, rent) and want to record reimbursements without losing the original line for bank reconciliation. Decision: separate `expense_payments` table 1:N to expense, original amount untouched, all aggregates use `effective = amount − SUM(payments)`. Sum of payments must stay strictly less than the expense amount (Spendio is not an income tracker). Payments are private to the expense creator even on shared sheets. See PRD §5.

**Trade-offs accepted** (chosen vs. rejected): separate table over columns on `expenses` (one extra JOIN/SUM per query, clean schema); 1:N over 1:1 (modal with list + aggregate validation); single `description` text over a structured `shared_with` (cannot filter "everything shared with X" without parsing); required description in UI over optional (small friction, readable data long-term); no settled/pending status à la Splitwise (no debt tracking, only what already moved); strict `SUM < expense` over permissive-with-warnings (keeps Spendio out of income territory).

**Data model** (in `server/schema.ts`):

- [ ] Create table `expense_payments`:
  - [ ] `id INTEGER PRIMARY KEY AUTOINCREMENT`.
  - [ ] `expense_id INTEGER NOT NULL REFERENCES expenses(id) ON DELETE CASCADE`.
  - [ ] `amount REAL NOT NULL CHECK (amount > 0)`.
  - [ ] `description TEXT` (nullable in DB, **required in UI**).
  - [ ] `user_id INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE`.
  - [ ] `created_at TEXT NOT NULL DEFAULT (datetime('now'))`.
  - [ ] `updated_at TEXT NOT NULL DEFAULT (datetime('now'))`.
- [ ] Index `idx_expense_payments_expense_id` on `expense_id`.
- [ ] Trigger `expense_payments_updated_at` to refresh `updated_at` on row update.

**Backend validation** (lives in service layer, not DB — SQLite `CHECK` cannot read other tables):

- [ ] On create payment: `amount > 0` and `SUM(payments) + new.amount < expense.amount`.
- [ ] On edit payment: same rule, replacing the affected payment in the SUM.
- [ ] On delete payment: no restriction.
- [ ] On edit `expense.amount` (existing `PUT /api/expenses/:id`): reject if `new_amount ≤ SUM(payments)` with a clear error message.
- [ ] Cascade-delete payments when the parent expense is deleted (covered by FK).

**API:**

- [ ] `GET /api/expenses`: each expense returns pre-aggregated `shared_total` plus its `payments[]`. Single query with aggregated JOIN — no N+1 when rendering the table.
- [ ] `POST /api/expenses/:id/payments`: create payment, applies SUM rule.
- [ ] `PUT /api/expenses/:id/payments/:paymentId`: edit payment, applies SUM rule.
- [ ] `DELETE /api/expenses/:id/payments/:paymentId`: delete payment.

**Frontend** (anchors: `src/components/expenses/ExpenseRow.tsx`, `src/components/expenses/ExpensesPage.tsx`, `src/api/expenses.ts`):

- [ ] `getEffectiveAmount(expense)` helper, single source of truth, consumed by monthly total, category breakdown and charts.
- [ ] `ExpenseRow`: large original amount + small effective amount when payments exist + a "shared" visual indicator + row-menu action to open the modal.
- [ ] `ExpensePaymentsModal`: list of payments (amount, description, delete) + `+ Add payment` button + footer `Total shared: X / Y · Real cost: Z`. Description required at submit.
- [ ] i18n strings for the modal, tooltips and validation errors (es primary).

**Open product/UX questions** (resolve before or during implementation):

- [ ] Visual "shared" indicator on the row: icon, color, badge, payment count? UX polish.
- [ ] Default payment list ordering inside the modal: by date, by amount, or manual?
- [ ] Exact wording of the error when blocking `expense.amount` edits below the SUM.
- [ ] Visibility on shared sheets — confirm payments stay private to the expense creator (recommended in PRD §5.3).
