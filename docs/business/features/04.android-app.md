# ANDR — Android app

Native Android client (Kotlin + Jetpack Compose) consuming the same Express API as the web. Single source of truth in the backend; no offline cache in the MVP. The Android project lives in `android/` inside this monorepo. All work is on the backlog — nothing shipped yet.

## Rules

### Architecture
- The backend is the single source of truth. Both clients (web + Android) see the same data through the same API. Changes from one client are immediately visible to the other.
- No client-side persistence beyond `DataStore` (server URL, API key, theme preference). MVP fetches fresh on every screen change; offline cache via Room is deferred.
- Domain logic that already lives in the web `src/helpers/*` (totals, MoM comparison, category grouping, currency/date format, Excel export) is **ported, not re-derived** — Kotlin equivalents must produce the same outputs for the same inputs.

### Auth & deployment
- Authentication for Android is a static `X-API-Key` header. Suitable for personal LAN/tunnel use; not for public deployments. No OAuth/JWT.
- The backend stays backward-compatible: if `API_KEY` env var is unset, the middleware lets every request through (today's web flow is untouched).
- The backend binds to `0.0.0.0` so it accepts LAN connections. Deployment options: same WiFi or a tunnel (Tailscale / ngrok). No cloud hosting needed.
- Server URL and API key are configured in-app at runtime — never hardcoded in the APK.

### UI
- Material3 design system. Theme respects system dark/light and Android 12+ dynamic color when available.
- Currency and date formatting match the web: EUR (`es-ES`).

## Out of scope (MVP)

- Offline-first behavior / Room cache (deferred to a later phase).
- iOS app.
- Multi-account on the same device.
- Push notifications.
- Cloud-hosted backend; this app targets personal LAN/tunnel deployments.
- OAuth or JWT-based auth (not justified for a personal client).
