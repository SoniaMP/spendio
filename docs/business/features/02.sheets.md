# SHT — Sheets

Sheets are the top-level container for expenses (e.g. "Personal", "Lisbon trip"). Owners create, edit, reorder and delete sheets, switch between them, and optionally share a sheet with another user with read or edit permission.

## Rules

- Every expense belongs to exactly one sheet.
- Sheets are owned by a single user; ownership is not transferable.
- Sharing is granted per sheet, with permission `read` or `edit`. Other sheets stay private.
- When a sheet is shared with an email that has no account, a stub user is created and activated via the password-reset flow (see [`features/00.authentication.md`](00.authentication.md) UC-AUTH-06).
- Invited users can set a custom display name for a shared sheet that only affects their own view.
- Payments on expenses (when the feature ships) are private to the expense creator, even on shared sheets.

## Backlog

### Auto-activation email when sharing with a stub

Today, when a sheet is shared with a non-existing email, the stub user only finds out if the owner tells them and asks them to use the password-reset flow. We want to send an activation email automatically at invite time so the invited person has a clear path in. The implementation lives in mailing — see [`features/03.mailing.md`](03.mailing.md) backlog block "Auto-activation email when sharing a sheet with a stub". This block stays here to flag the UX gap; once shipped, UC-SHT-07 should be updated to reference the auto-trigger.

### Revoke shared sheet access

Owners can grant access via UC-SHT-06 but there is no flow yet to revoke it — once shared, access is permanent. When this lands, the revoke action must cascade into `recurring_expenses` (see [`features/01.expenses.md`](01.expenses.md) UC-EXP-16): every template owned by the revoked user on that sheet must be auto-deactivated synchronously, not waiting for the cron's safety net to catch it on the next run. Otherwise recurrences silently stop generating and the user has no signal as to why.

- [ ] UI on the share dialog to remove a previously granted access.
- [ ] Endpoint to revoke a user's access to a sheet.
- [ ] On revoke, set `is_active = false` on every `recurring_expenses` row where `user_id = revoked_user` AND `sheet_id = sheet`.
- [ ] Decide whether to notify the revoked user that their recurrences on that sheet were paused (channel TBD).
