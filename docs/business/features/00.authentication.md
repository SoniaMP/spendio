# AUTH — Authentication

Login, registration and password management for owner and stub users. Supports Google OAuth and email + password, with a unified password-reset flow that also activates stub accounts created by sheet sharing.

## Rules

- Email matching is case-insensitive at lookup and at unique-constraint level.
- The system never reveals whether an email exists (no enumeration via login or reset endpoints).
- Password-reset links expire after 30 minutes and are single-use.
- Password-reset requests are rate-limited to 5 per hour per email.
- A stub user (created by sheet sharing) is activated through the same reset-password flow — no separate "activate" endpoint.
- Dev login is only available when `VITE_AUTH_BYPASS=true`. Never enable in production builds.

## Use cases

<details>
<summary>

### UC-AUTH-01 — Login with Google OAuth · DONE

</summary>

- **Status:** DONE
- **Actor:** Any user
- **Trigger:** User clicks "Sign in with Google" on the login page.

**Main flow**
1. User is redirected to the Google OAuth consent screen.
2. On success, the backend resolves or creates the user by Google email.
3. A session cookie is issued and the user lands on the active sheet.

</details>

<details>
<summary>

### UC-AUTH-02 — Login with email + password · DONE

</summary>

- **Status:** DONE
- **Actor:** Registered user
- **Trigger:** User submits the email + password form.

**Main flow**
1. Backend looks up the user by lowercased email.
2. Password hash is verified.
3. Session cookie is issued.

**Alternative flows**
- **2a. Wrong credentials:** generic "invalid credentials" error, no hint about which field failed.

</details>

<details>
<summary>

### UC-AUTH-03 — Register a local account · DONE

</summary>

- **Status:** DONE
- **Actor:** New user
- **Trigger:** User submits the registration form (email + password).

**Main flow**
1. Email is normalized to lowercase and checked for uniqueness.
2. Account is created with hashed password.
3. 8 default categories are seeded for the new user.
4. User is logged in.

</details>

<details>
<summary>

### UC-AUTH-04 — Request password reset · DONE

</summary>

- **Status:** DONE
- **Actor:** Registered or stub user
- **Trigger:** User submits the "forgot password" form with their email.
- **Refs:** MAIL

**Main flow**
1. If the email exists, a single-use token (30 min TTL) is generated and sent via Resend.
2. The response is identical regardless of whether the email exists (anti-enumeration).
3. Rate limit: max 5 requests per hour per email.

</details>

<details>
<summary>

### UC-AUTH-05 — Set new password via reset link · DONE

</summary>

- **Status:** DONE
- **Actor:** User holding a valid reset token
- **Trigger:** User opens the reset link and submits a new password.
- **Refs:** MAIL

**Main flow**
1. Token is validated (exists, not expired, not consumed).
2. Password is hashed and stored.
3. Token is invalidated.
4. User is redirected to login.

**Alternative flows**
- **1a. Stub user:** same flow; the password set here also activates the account for sheet access (UC-AUTH-06 is the same flow seen from the stub side).

</details>

<details>

### UC-AUTH-06 — Activate stub account · DONE

</summary>

- **Status:** DONE
- **Actor:** Stub user (created by sheet sharing)
- **Trigger:** Stub user submits the "forgot password" form.
- **Refs:** SHT, MAIL

**Main flow**
1. Same as UC-AUTH-04 + UC-AUTH-05.
2. After setting password, the user can log in and access the sheets they were invited to.

## Backlog

### Session timeout with pre-expiry warning

Sessions currently last 7 days. We want to expire them after 1 hour of inactivity, with a 5-minute heads-up so the user can renew before losing work. Decision: rolling cookie on backend (resets the inactivity timer on every request), frontend hook tracks user activity events and surfaces a warning dialog before forcing logout.

Constants (shared or duplicated backend/frontend): `SESSION_MAX_AGE = 60 * 60 * 1000` (1 h), `WARNING_BEFORE_MS = 5 * 60 * 1000` (5 min).
